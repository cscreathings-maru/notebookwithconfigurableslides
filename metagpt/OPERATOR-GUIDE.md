# Operator Guide — Building the platform with MetaGPT

How to go from the Spec Kit artifacts to working code using MetaGPT, slice by slice, with review
and integration in the loop. Read this once before you start.

## Mental model

- **Spec Kit** (this project) owns the *what/why/governance*: `constitution.md`, `spec.md`,
  `plan.md`, `research.md`, `data-model.md`, `contracts/`, `tasks.md`.
- **MetaGPT** is a code generator you run **per slice** using the prompts in `metagpt/prompts/`.
  It produces a repo in `./workspace`; you review it against the contracts and integrate the parts
  you want (the orchestrator backend and frontend — never the engines).
- **You + Cowork** review, test, integrate, and enforce the constitution (MetaGPT won't).

MetaGPT is an accelerator, not an autopilot. Expect to keep ~60–80% of generated code per slice
and rework the rest at the integration seams.

> **Using the multi-model loop (Claude plans, DeepSeek codes)?** That is the recommended path —
> see `../automation/AUTOMATION-PIPELINE.md`. There, **Aider** (architect=Claude, editor=DeepSeek)
> is the inner-loop orchestrator and MetaGPT is optional first-draft scaffolding. The slice order
> and gates below still apply.

## One-time setup

1. On your machine (not the app sandbox), install prerequisites: Python 3.9–3.11, Node.js, pnpm.
2. Run `bash metagpt/install.sh` (creates a conda env, installs MetaGPT, drops a config).
3. Edit `~/.metagpt/config2.yaml` — set your **DeepSeek** key (template in `config2.example.yaml`).
   Use `deepseek-chat` for codegen; switch to `deepseek-reasoner` for the harder design slices
   (slice 0 and 3). Keep temperature low (~0.2).
4. Create a code repo for the build, e.g. `notebookllm-platform/`, with `backend/`, `frontend/`,
   `deploy/`. Copy `deploy/docker-compose.yml` and `deploy/.env.example` there.

## Run a slice

From a scratch codegen directory (MetaGPT writes to `./workspace`):

```bash
conda activate metagpt
metagpt "$(cat /path/to/metagpt/prompts/slice-0-platform-skeleton.md)"
# output lands in ./workspace/<project>
```

Then:
1. **Review** against `contracts/orchestrator-api.md`, `contracts/engine-integration.md`,
   `data-model.md`, and the matching phase in `tasks.md`.
2. **Integrate** the backend/frontend code into your repo (discard anything that tries to
   reimplement Open Notebook or Presenton).
3. **Run tests**; the prompts ask MetaGPT to write tests-first — make them pass.
4. **Commit** on a per-slice branch.

Iterate within a slice using MetaGPT's incremental mode (check your version's docs, e.g. an
`--inc` flag) or just re-prompt with a tighter, smaller follow-up.

## Slice order (each is independently shippable)

| Order | Prompt | Outcome | Gate before moving on |
|-------|--------|---------|------------------------|
| 1 | slice-0-platform-skeleton | compose stack up, OIDC login, tenancy + RBAC, job framework | cross-tenant test returns 404; viewer blocked |
| 2 | slice-1-ingestion | upload → Open Notebook analysis → status | a PDF reaches `ready`; bad file flagged |
| 3 | slice-2-registry | versioned profiles + templates, PPTX import | used version is immutable |
| 4 | slice-3-generation | outline contract → Presenton → consistency check | same inputs → same section structure; PPTX+PDF download |
| 5 | slice-4-refine-provenance | edit outline, regenerate w/o re-ingest, history | regenerate doesn't re-ingest; full provenance shown |
| 6 | slice-5-usage-audit | metering, quotas, audit dashboard | rollups correct; quota blocks |

After slice 3 you have a working MVP (the core value loop). Slices 4–5 are enhancements.

## Bring the runtime up

The engines and infra come from `deploy/docker-compose.yml` (they are NOT generated by MetaGPT):

```bash
cd deploy && cp .env.example .env   # fill in keys
docker compose up -d
docker compose exec orchestrator alembic upgrade head
```

See `specs/001-presentation-notebook-llm/quickstart.md` for the smoke test and isolation check.

## Guardrails (enforce after every slice)

- **Engines stay internal** — no published ports for `open-notebook`/`presenton`; only the
  orchestrator holds their URLs/creds (compose already does this).
- **Tenant isolation** — every query filtered by `tenant_id`; cross-tenant returns 404. Keep the
  contract test green.
- **Deterministic output** — structure comes from the validated outline, not free-form LLM text;
  profiles/templates are versioned and immutable once used.
- **Tests + evals gate merges** — contract tests pinned to engine versions; the consistency eval
  must pass.
- **Provenance + metering** — every generation records sources, profile/template version, model,
  tokens/cost.

## What Cowork can do for you next

- Generate slice prompts for any *additional* features beyond the MVP.
- Review MetaGPT's generated code against the contracts and the constitution, and list gaps.
- Write/repair tests, wire integration seams, and debug failures.
- Produce the Alembic migrations, seed scripts, or CI config if MetaGPT's are thin.
- Keep the Spec Kit docs in sync as scope evolves (new spec → plan → tasks).

## Cost & expectations

DeepSeek is inexpensive, but multi-role runs still add up — generate per slice, not all at once.
Treat MetaGPT output as a strong first draft: keep the structure, harden the security/isolation and
the engine integration yourself, since those are exactly where generic codegen is weakest.
